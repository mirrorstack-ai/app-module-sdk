# Contributing to @mirrorstack-ai/app-module-sdk

## Before you start

**Important:** Before forking and starting work on an issue, you **must** leave a comment on the issue explaining:

1. **What** you want to implement
2. **How** you plan to implement it (approach, key decisions)
3. Wait for a maintainer to approve your approach

This prevents wasted effort, duplicate work, and ensures alignment with the SDK design. PRs without a prior approved comment will be closed.

## Getting started

```bash
git clone git@github.com:mirrorstack-ai/app-module-sdk.git
cd app-module-sdk
go mod tidy
go test ./...
```

Requires Go 1.26+.

Questions? Join us on [Slack](https://join.slack.com/t/mirrorstackai/shared_invite/zt-3twmj15cm-EPfQscE71I~JJj0yHK6EZg).

## Branch naming

```
feat/SDK-{number}_{slug}       New feature or package
refactor/SDK-{number}_{slug}   Restructuring existing code
fix/SDK-{number}_{slug}        Bug fix
chore/SDK-{number}_{slug}      Deps, config, CI
```

Append `/agent` when the work is done by an AI agent:

```
feat/SDK-{number}_{slug}/agent
```

Examples: `feat/SDK-1_init-and-start`, `fix/SDK-5_lambda-payload-parsing/agent`

## Labels

| Category | Labels |
|----------|--------|
| Work | `feature` `bug` `test` `docs` `refactor` `chore` |
| Priority | `p0:critical` `p1:high` `p2:medium` `p3:low` |
| Size | `size:xs` `size:s` `size:m` `size:l` |
| Status | `blocked` `needs triage` |
| Contributor | `good first issue` `help wanted` |

Look for issues labeled [`good first issue`](https://github.com/mirrorstack-ai/app-module-sdk/issues?q=is%3Aopen+label%3A%22good+first+issue%22) to get started.

## PR workflow

1. Comment on the issue with your implementation plan
2. Wait for maintainer approval
3. Fork and create a branch with the naming convention above
4. Implement with tests
5. Push and open a PR referencing the issue (`Closes #N`)
6. PR requires maintainer review before merge
7. Squash merge to `main`

## Go conventions

- Module path: `github.com/mirrorstack-ai/app-module-sdk`
- Go 1.26+
- Use [chi v5](https://github.com/go-chi/chi) for routing
- Tests alongside source files (`foo_test.go`)
- Run `go vet ./...` and `go test ./...` before pushing
- No third-party dependencies without discussion — keep the SDK lightweight

## Reference implementation

The `reference/v2-restored` branch contains the original implementation with handler, event, meter, and storage packages (with tests). This is being restructured into the new configless design.

Use it as reference for understanding the domain, but don't copy the API surface — the new design is different. See the README roadmap for the target API.

## Architecture decisions

Key design decisions are documented in the protocol spec (shared internally). The highlights:

- **Configless** — no YAML, code is the single source of truth
- **`ms.Init()` + `ms.Start()`** — single entry point, auto-detects Lambda vs HTTP
- **Auth scopes via `ms.Platform()` / `ms.Public()` / `ms.Internal()`** — not directory-based magic
- **`/__mirrorstack/*`** — system routes auto-generated by the SDK
- **`description` and `category` are platform-only** — modules don't self-describe to prevent prompt injection

If you have questions about design decisions, ask in the issue or on Slack before implementing.
